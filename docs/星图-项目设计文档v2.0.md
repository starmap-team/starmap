## 一、项目概述

### 1.1 项目背景

在国家从"人口红利"向"人才红利"转型的关键期，新一代信息技术领域（人工智能、大数据、物联网等）技术迭代速度远超人才培养周期，导致劳动力市场出现严重的结构性矛盾——企业面临"人岗匹配度低、新兴岗位识别难"的困境，求职者则缺乏动态的行业技能指引。

**星图(StarMap)** 项目聚焦"数据驱动+大模型+知识图谱"核心技术方向，构建一个可自我进化的"人才能力大脑"，实现从"静态画像"向"动态感知"的跨越。

### 1.2 项目定位

**星图 —— 人才能力星云导航系统**，寓意为每一颗"人才之星"在职业星云中提供精确的导航。系统通过多源异构数据的采集、融合与智能分析，构建新一代信息技术领域岗位能力知识图谱，支持岗位能力的动态演化追踪，并为求职者提供精准的人岗匹配诊断与个性化学习路径规划。

### 1.3 核心价值主张

| 价值维度 | 目标用户 | 核心价值 |
|---------|---------|---------|
| 岗位发现 | 企业HR、行业分析师 | 自动识别新兴岗位，生成标准化岗位定义 |
| 能力追踪 | 企业HR、培训管理者 | 实时追踪既有岗位的能力要求变化 |
| 全景导航 | 求职者、在校学生 | 一览新一代信息技术领域的岗位与技能全景 |
| 精准匹配 | 求职者 | 解析简历技能，诊断与目标岗位的差距 |
| 路径规划 | 求职者、在校学生 | 基于技能差距生成个性化学习路径 |

### 1.4 项目目标

#### 1.4.1 量化指标（可验证）

| 指标类别 | 指标项 | 目标值 | 验证方式 |
|---------|--------|--------|---------|
| 图谱规模 | 岗位覆盖数 | >= 30个 | Cypher COUNT 查询 |
| 图谱规模 | 标准化技能节点 | >= 500个 | Cypher COUNT 查询 |
| 准确率 | JD解析准确率（F1） | >= 90% | 100条标注JD逐字段评估，见第十四章 |
| 准确率 | 简历提取准确率（F1） | >= 90% | 50份标注简历逐字段评估，见第十四章 |
| 准确率 | 人岗匹配准确率 | >= 90% | 100个(JD,简历)配对评估，见第十四章 |
| 测试 | JD及配套测试用例 | >= 100条 | Golden Set 验收（Kappa>=0.7）|
| 测试 | 单元测试覆盖率 | >= 60% | pytest-cov 报告 |
| 部署 | 容器化一键部署 | Docker Compose | `docker-compose up` 全服务启动 |

#### 1.4.2 "≥90%准确率"的可验证拆解

为避免"准确率达标但评委不认可"的风险，三项准确率必须按以下口径量化（详细方法学见第十四章）：

| 指标 | 拆解维度 | 子指标 |
|------|---------|--------|
| **JD解析准确率** | 按字段分别评估 | job_title F1 / required_skills F1 / bonus_skills F1 / experience F1 / education F1，加权汇总 |
| **简历提取准确率** | 技能规范化容忍 | 技能经归一化（别名映射+向量匹配阈值0.85）后比对；含/不含熟练度两层评估 |
| **匹配准确率** | 判定协议 | 主协议：阈值二元判定（match_score>=0.6为匹配）的准确率；辅报告：Top-K命中率、Kendall Tau 相关性 |

> ⚠️ **关键声明**：技能匹配采用"规范化后精确匹配 + 模糊匹配标记为 partial"两层判定，partial 不计入正确也不计入错误，避免虚高或虚低。详见 §14.3。

#### 1.4.3 非目标声明（明确不做，防止范围蔓延）

为聚焦核心闭环与可验证性，以下**不在本期范围内**：

- ❌ 不做全国/全行业岗位覆盖，仅聚焦新一代信息技术领域（AI/大数据/智能系统/物联网）
- ❌ 不做实时流式数据管道，采用批处理 + 定时演化分析
- ❌ 不做移动端App，仅交付Web端
- ❌ 不做账号体系与多租户，评审环境单租户即可
- ❌ 不做全量历史数据回溯（受3个月项目周期限制，时序演化采用真实+合理推演混合，见 §16.2）
- ❌ 不自研LLM，全部基于讯飞星火API + Qwen2.5本地模型

## 二、赛题需求分析

### 2.1 四大核心功能模块

**模块A — 新岗位发现与定义**

| 约束 | 支持人工优化与动态更新 |

| 属性 | 描述 |
|------|------|

| 输出 | 能力项增/删/改，附更新说明及数据源 |

**模块C — 新一代信息技术岗位全景图谱**

| 属性 | 描述 |
|------|------|

### 3.2 各层职责详述

#### L1 基础设施层

提供系统运行的基础支撑环境：

- **Docker Compose容器编排**：统一管理所有服务组件的生命周期
- **配置中心**：基于YAML的环境配置管理，支持开发/测试/生产环境切换
- **API网关**：统一入口，负责路由分发、限流、认证
- **日志监控**：结构化日志 + 健康检查接口

#### L2 数据融合层

处理**多源异构**数据的采集、清洗与融合。本层是赛题"多源异构"要求的核心承载，需显式融合三类异构数据，而非仅做"多源JD爬取"。

##### 三类异构数据源

> ✅ **决策记录（D5）**：采用**最小可行档3源**，做实真正的异构融合而非纸面覆盖。三类结构类型（结构化/半结构化/非结构化）各取一个代表源，构成真正的异构融合。其余源（O*NET、人社部大典、GitHub趋势等）作为"架构可扩展"声明写入文档，不在本期实现。理由：评委要的是"能证明真的融合了异构数据"，3源已满足且每类都能做深；贪多会导致每类都浅。

| 数据类型 | 本期接入源 ✅ | 结构特征 | 融合处理策略 | 可扩展源（声明，不实现）|
|---------|-----------|---------|-------------|---------------------|
| **结构化** | **ESCO 技能分类标准**（公开API/数据集） | 字段固定、Schema明确 | 直接映射到本体节点，作为图谱的"权威锚点" | O*NET、人社部《国家职业分类大典》|
| **半结构化** | **BOSS直聘 + 拉勾 + 猎聘 JD**（HTML爬取） | 字段半定、含噪声 | 爬取→清洗→LLM抽取→归一化→信任度评分 | 智联招聘、企业官方招聘站 |
| **非结构化** | **掘金技术热榜**（技术博客） | 自由文本、含隐含技能 | 全文抽取→技术名词NER→时序频率统计→辅助验证 | CSDN、GitHub趋势、课程大纲、行业报告PDF |

##### 异构数据对齐机制（核心创新点承载）

不同源抽取出的技能/岗位需对齐到统一本体，避免"各源各表"：

```
结构化源（ESCO "机器学习")    ─┐
半结构化源（JD "ML算法")       ├─► 技能归一化引擎 ─► 统一节点 [机器学习]
非结构化源（博客 "Machine Learning") ─┘     (向量相似度+别名映射)

对齐成功 → 合并证据，累加 TrustScore
对齐失败 → 标记 [NEW]，进入多源验证流程（≥3源确认才入图）
```

##### 子组件职责

- **数据采集器**：Scrapy + Playwright爬虫框架（半结构化JD）；PDF解析器+API对接器（结构化/非结构化）
- **清洗管道**：基于规则引擎的文本标准化、格式统一、HTML标签剥离
- **去重引擎**：SimHash + 编辑距离的JD级去重（解决"抄袭"）
- **归一化引擎**：跨源技能/岗位别名映射（如"Python"="python3"="Python编程"="py"），见 §6.2
- **数据信任度评估器**：多维度计算数据质量评分，是异构融合的"裁判"，见 §7.1

#### L3 数据服务层

封装各类数据存储的访问接口：

- **图数据服务**：Neo4j图数据库的CRUD封装，Cypher查询优化
- **关系数据服务**：PostgreSQL存储JD原始数据、用户数据、变更日志
- **向量检索服务**：Chroma存储技能/岗位嵌入向量，支持语义相似度检索

## 五、核心功能模块详细设计

### 5.1 模块A：新岗位发现与定义

#### 业务流程

```
多源数据采集 -> 数据清洗去重 -> 技能聚类分析 -> 新兴岗位候选发现
                                                    |
时序频率统计 -> 技能组合模式识别 -> LLM生成岗位定义 -> 幻觉防控校验
                                                          |
                                                 人工审核确认 -> 入图谱
```

#### 技术实现

**Step 1 — 新兴技能涌现检测**

基于Z-score异常检测识别涌现技能：
- 统计各技能在滑动时间窗口内的出现频率
- 计算Z-score = (近30天平均频率 - 历史基线均值) / 历史基线标准差
- Z-score > 2.0 的技能标记为"涌现中"

**Step 2 — 技能聚类 -> 候选岗位发现**

- 基于涌现技能的共现矩阵，使用HDBSCAN聚类
- 每个聚类代表一个"候选新岗位"的技能组合
- 过滤条件：聚类中技能数 >= 3，且在至少5个独立来源中出现

**Step 3 — LLM生成岗位定义**

通过讯飞星火API，使用结构化Prompt生成包含5要素的岗位定义：
- 岗位名称
- 核心职责（3-5条）
- 必备技能（从聚类中提取）
- 加分技能
- 典型行业应用场景

**Step 4 — 幻觉防控校验**

生成的岗位定义需通过三层防线（详见第七章创新点2）

#### 新岗位发现演示策略 ⭐ v2.0新增

赛题要求演示"新岗位发现"全过程，且强调真实性（非预设）。需准备完整证据链，确保评委确信岗位是"被系统发现"而非"被人工预设"：

**演示目标新岗位**：大模型应用工程师 ✅（D2 决策确定）| **种子数据已添加** ✅（2026-06-26，PR#42）

> 选定理由：数据量适中（≥30条JD易达成）、新颖度高（区别于已较普遍的AI Agent工程师）、与发榜单位科大讯飞（星火大模型）业务高度相关，评委代入感强。

**证据链必备要素**：

| 证据要素 | 具体内容 | 数据来源 |
|---------|---------|---------|
| ① 涌现检测信号 | 该技能组合的 Z-score > 2.0，30天内频率同比上涨 X% | 时序统计 |
| ② 多源独立确认 | 在 >=5 个独立企业来源中出现 | 跨源去重后JD集合 |
| ③ 技能聚类证据 | HDBSCAN聚类产生独立簇，与现有岗位 Jaccard < 0.4 | 聚类分析 |
| ④ LLM生成的5要素定义 | 名称/职责/必备技能/加分技能/行业场景 | 讯飞星火API |
| ⑤ 幻觉防控通过 | 三层防线全部通过，置信度 >= 0.8 | 防控引擎 |
| ⑥ 人工审核记录 | 审核人、审核意见、最终入图时间 | capability_changelog |

**演示数据准备**（详见 §16）：
- 准备 >=30 条真实的"AI Agent"相关 JD 作为输入
- 演示时从原始 JD 一路展示到图谱节点，每个环节可追溯
- 保留中间产物（聚类结果、抽取JSON、验证报告）作为附件

### 5.2 模块B：既有岗位能力动态更新

#### 业务流程

```
定期采集目标岗位最新JD -> 批量信息抽取 -> 与图谱现有能力项对比
                                                        |
                              生成变更提案(增/删/改) -> 信任度评分
                                                        |
                                      人工审核队列 -> 确认后更新图谱
```

#### 差分检测算法

对同一岗位的新旧技能集合进行集合运算：

```
新增技能 = 新抽取技能集 - 图谱现有技能集
删除技能 = 图谱现有技能集 - 新抽取技能集
修改技能 = 技能要求程度或权重发生变化的交集部分
```

#### 变更提案结构示例

```json
{
  "position": "Java开发工程师",
  "snapshot_date": "2026-07-01",
  "changes": [
    {
      "type": "add",
      "skill": "Spring AI",
      "evidence": {
        "source_count": 8,
        "first_seen": "2026-05-15",
        "trust_score": 0.82,
        "sample_jds": ["JD_ID_001", "JD_ID_042"]
      }
    },
    {
      "type": "remove",
      "skill": "JSP/Servlet",
      "evidence": {
        "decline_rate": -0.65,
        "source_count": 2,
        "trust_score": 0.71
      }
    }
  ]
}
```

### 5.4 模块D：人岗匹配度诊断与差距分析

#### 匹配诊断全流程

```
用户上传简历(PDF/Word) -> 文档解析 -> LLM技能提取 -> 技能归一化
                                                          |
                        学习路径生成 <- 差距分析报告 <- 与目标岗位技能对比
                              |

```

#### 简历解析Pipeline

1. **文档解析**：pdfplumber(PDF) + python-docx(Word) -> 纯文本
2. **LLM结构化抽取**：讯飞星火API -> JSON格式技能列表
3. **技能归一化**：向量相似度匹配 + 别名映射表 -> 标准技能名
4. **准确率保障**：双模型交叉验证（星火API + Qwen2.5本地），不一致项人工校验

#### 简历格式健壮性处理 ⭐ v2.0新增

赛题要求"解析简历（PDF/Word）"，但真实简历格式多样，单一解析路径会显著降低简历提取准确率。需分级处理：

| 格式场景 | 检测信号 | 处理策略 |
|---------|---------|---------|
| **文本型PDF/Word** | pdfplumber直接提取文本长度正常 | 走标准Pipeline（上述4步）|
| **扫描件/图片PDF** | 提取文本为空或极少 | OCR降级：调用OCR引擎（如PaddleOCR）→ 文本后再走Pipeline；标记 `is_ocr=true` |
| **多栏/表格简历** | 提取文本顺序混乱 | 启用表格检测模式：保留行列结构后再抽取；优先用python-docx的表格API |
| **加密/损坏文件** | 解析报错 | 降级为"解析失败，请手动输入技能"，引导用户走模块D的手动输入路径 |
| **超长简历**（>10页）| 文本超LLM上下文窗口 | 分段抽取 + 结果合并去重 |

**降级流程图**：

```
简历文件上传
    │
    ├─ 文本提取成功 ──► 标准LLM抽取 ──► 归一化 ──► 结果
    │
    ├─ 文本极少/为空 ─► OCR引擎 ──┐
    │                              ├─► 合并文本 ─► LLM抽取 ─► 结果
    ├─ 检测到表格  ─────────────► 表格API ┘
    │
    └─ 解析失败 ──► 提示用户手动输入技能
```

> ⚠️ **准确率风险提示**：OCR降级路径的准确率会低于标准路径，评估时（§14）需对 OCR 样本单独报告，避免拉低整体指标或虚高。

### 6.2 技能归一化算法

三步归一化策略：

**Step 1 — 精确别名映射**：查表将已知别名映射为标准名（如"python3" -> "Python"）

**Step 2 — 向量语义匹配**：对别名表中未覆盖的技能名，计算BGE嵌入向量，在Chroma中检索最相似的标准技能名，相似度阈值0.85

**Step 3 — 新技能标记**：无法匹配到现有技能的标记为[NEW]，进入多源验证流程

### 6.3 双路召回匹配引擎

```
用户技能集合 S = {s1, s2, ..., sn}
目标岗位技能集合 T = {t1, t2, ..., tm}

路径1 — 精确匹配（技能名归一化后集合运算）
  exact_match = S 与 T 的交集
  exact_score = |exact_match| / |T|

路径2 — 语义匹配（向量相似度）
  对T中每个未精确匹配的技能ti:
    semantic_score_i = max(cosine_sim(embed(ti), embed(sj)) for sj in S)
  semantic_match_count = T中未被精确匹配但语义匹配度>0.8的技能数
  semantic_score = semantic_match_count / |T - exact_match|

综合匹配分数：
  match_score = 0.7 * exact_score + 0.3 * semantic_score

必备/加分技能加权：
  final_score = required_match_score * 0.7 + bonus_match_score * 0.3
```

### 6.4 数据清洗与去重算法

**SimHash去重**：

1. 对JD文本分词后计算SimHash指纹（64位）
2. 两条JD的SimHash汉明距离 < 阈值（默认10）视为疑似重复
3. 重复组内保留最早发布的版本（时效性优先）

**技能归一化映射**：

构建别名映射表，覆盖常见技能表述变体：
- 语言类："python" / "python3" / "Python编程" -> "Python"
- 框架类："react.js" / "reactjs" / "React JS" -> "React"
- 概念类："机器学习" / "ML" / "Machine Learning" -> "机器学习"

### 6.5 学习路径生成算法 ⭐ v2.0补充

赛题要求"针对性改进建议与岗位学习路径规划"，学习路径依赖图谱的 PREREQUISITE（技能前置依赖）关系。该关系如何构建是关键——纯人工标注不可扩展，纯共现推断会有噪声，采用混合策略。

#### PREREQUISITE 关系的构建（混合策略）

**来源1 — 人工种子（高置信，覆盖核心技能）**

由领域专家手工标注核心技能的依赖链，作为"金标准"骨架：
```
深度学习 -> 机器学习 -> Python基础 -> 编程基础
          -> 线性代数
          -> 概率统计
TensorFlow -> 深度学习
            -> Python基础
K8s -> Docker -> Linux基础
```
目标：人工种子覆盖 >=100 条核心 PREREQUISITE 关系。

**来源2 — 共现方向性推断（中置信，需验证）**

从 JD 数据中，"要求 A 同时要求 B"的高频共现，按学习难度方向性推断依赖：
```
对高频共现技能对 (A, B)：
  若 A 在JD中的经验要求年数 系统性 > B 的年数
  且 共现次数 >= 20（统计显著）
  则推断 B 是 A 的前置（B -> A）
```
推断结果经 §7.2 幻觉防控校验后，置信度 >= 0.7 才入图。

**来源3 — 外部课程大纲对齐（辅助验证）**

抓取主流课程（如Coursera/慕课网）的课程大纲，技能出现顺序作为依赖信号：
```
若课程C的大纲中技能B章节在技能A章节之前出现
且 >=3 门独立课程一致
则增强 B -> A 的 PREREQUISITE 置信度
```

#### 学习路径生成（基于 PREREQUISITE 图的拓扑排序）

用户存在技能缺口 `G = {g1, g2, ...}`（目标岗位缺失技能集），生成学习路径：

```
对每个缺失技能 gi：
  1. 反向遍历 PREREQUISITE 关系链，找到 gi 的所有前置祖先节点
  2. 过滤掉用户已掌握的节点
  3. 得到"需补充的前置链" path_i
  4. 多个缺口的前置链取并集，做拓扑排序（保证前置先学）
  5. 按拓扑序标注每步的预估学习时长（从 LearningResource.duration 汇总）
  6. 输出带时间标注的学习路径
```

**输出示例**（用户缺口含"TensorFlow"）：
```
Step 1: Python基础 (1周) [前置缺口]
Step 2: NumPy (3天) [前置缺口]
Step 3: 线性代数基础 (1周) [前置缺口]
Step 4: 机器学习入门 (2周) [前置缺口]
Step 5: 深度学习基础 (2周) [前置缺口]
Step 6: TensorFlow入门与模型构建 (2周) [目标技能]
预估总时长：约8周（兼职学习）
```

## 七、创新点技术方案

### 7.1 创新点1：多源交叉验证的数据信任度模型

#### 核心公式

```
TrustScore(D) = w1 * Authority(D) + w2 * Timeliness(D) + w3 * Independence(D) + w4 * Consistency(D)
```

**Authority(D) — 来源权威性评分 [0, 1]**

| 来源类型 | 评分 |
|---------|------|
| 企业官方招聘网站 | 0.9 |
| 主流招聘平台（BOSS、拉勾、猎聘） | 0.7 |
| 猎头/聚合网站 | 0.5 |
| 社交媒体/论坛 | 0.3 |

**Timeliness(D) — 时效性评分 [0, 1]**

```
T(D) = exp(-2 * (now - publish_date).days / 180)
```
发布时间越近，时效性评分越高。超过6个月的JD时效性评分接近0。

**Independence(D) — 独立性评分 [0, 1]**

```
I(D) = 1 - max(SimHash_sim(D, D') for D' in same_source)
```
与同来源其他JD的最大相似度越低，独立性越高。高相似度意味着可能是"抄袭"。

**Consistency(D) — 一致性评分 [0, 1]**

```
C(D) = 交叉验证通过的技能数 / D中抽取的总技能数
```
技能需在>=2个独立来源的JD中出现才算"交叉验证通过"。

**默认权重**：w1=0.3, w2=0.2, w3=0.2, w4=0.3（一致性权重最高）

#### 信任度权重校准方法 ⭐ v2.0补充

上述默认权重不可任意设置，否则评委质疑。采用**标注数据驱动的网格搜索**校准：

**校准数据集**：从 Golden Set 中划出一个子集（~50条JD），人工标注每条JD的"真实可信度"标签（0-100，由双盲标注+仲裁确定）。

**校准流程**：
```
1. 定义权重空间：w1, w2, w3, w4 ∈ {0.1, 0.2, ..., 0.4}，且 w1+w2+w3+w4=1
2. 对每组权重组合，计算模型预测信任度 vs 人工标签的 Pearson 相关性
3. 选取相关性最高的权重组合作为最终配置
4. 报告校准结果（相关性系数、置信区间），写入评估报告
```

**校准验证**：在留出集（hold-out，未参与校准的样本）上验证最终权重的泛化性，防止过拟合。

**权重敏感性分析**：报告各维度权重 ±0.1 变动时，信任度排名的 Kendall Tau 变化，证明结果对权重不极端敏感。

#### 信任度驱动的图谱构建策略

| 信任度区间 | 标签 | 处理策略 |
|-----------|------|---------|
| [80, 100] | 高可信 | 直接入图谱 |
| [60, 80) | 较可信 | 入图谱，标记"待验证" |
| [40, 60) | 一般 | 不入图谱，进入人工审核队列 |
| [0, 40) | 低可信 | 丢弃或标记"高风险" |

#### 数据通胀检测

检测岗位JD中的技能要求"通胀"现象：
- 计算同岗位JD集合的技能数量分布
- 超过P90的JD标记为"疑似通胀"
- 以中位数作为"基准能力要求"
- 通胀指数 = 实际技能数 / 基准技能数

### 7.2 创新点2：基于本体约束的幻觉防控机制

#### 三层防线架构

```
                    LLM抽取结果
                         |
                    +----v----+
                    | 防线1   |  本体白名单校验
                    |Whitelist|  -> 已知技能库精确匹配
                    +----+----+  -> 向量语义匹配(阈值0.85)
                         | 未匹配项
                    +----v----+
                    | 防线2   |  多源交叉验证
                    |MultiSrc |  -> 新技能需在>=3个独立来源中出现
                    +----+----+  -> 时序连续性检查（非瞬时出现）
                         | 仍存疑项
                    +----v----+
                    | 防线3   |  置信度分级
                    |ConfGrade|  -> 标记为[已验证/待确认/高风险]
                    +----+----+  -> 高风险项不进入图谱
                         |
                    +----v----+
                    | 人工审核 |  待确认项由人工复核
                    |HumanRev |
                    +---------+
```

#### 置信度评分算法

```
confidence = 0.6 * multi_source_score + 0.4 * consistency_score

multi_source_score = min(独立来源数 / 3.0, 1.0)
consistency_score = 模型一致同意次数 / 总抽取次数

置信度分级：
  >= 0.8  -> "verified"（已验证）
  >= 0.5  -> "pending"（待确认）
  < 0.5   -> "high_risk"（高风险，不进入图谱）
```

#### RAG检索增强校验

对LLM抽取的每个技能，通过Chroma向量库检索已验证图谱中的最相似技能：
- Top-1相似度 > 0.9：确认为已知技能，直接归一化
- Top-1相似度在 [0.7, 0.9)：可能是已知技能的变体，标记为"待确认"
- Top-1相似度 < 0.7：可能是全新技能，进入多源验证流程

#### LLM-as-judge 幻觉自评估 ⭐ v2.0补充

传统三层防线是基于"外部规则"防控幻觉，但LLM自身的抽取合理性仍难自动评估。引入**LLM-as-judge**作为第四层自动化评估环，提升幻觉防控的可信度，并在评审中体现技术突破。

**评估机制**：
```
抽取结果（技能S + 抽取自JD段落P）
        │
        ▼
[裁判LLM] 输入：JD段落P + "请判断技能S是否真的被这段JD要求"
        │
        ├─ 判定 SUPPORTED（有证据支持）→ 通过
        ├─ 判定 UNSUPPORTED（无证据，疑似幻觉）→ 标记 high_risk
        └─ 判定 AMBIGUOUS（模糊）→ 进入多源验证
```

**裁判模型选型**：使用与抽取模型**不同的**模型族作为裁判（抽取用星火API，裁判用Qwen2.5本地），避免同源模型的系统性偏差。

**抽样审计**：LLM-as-judge 对每条JD的全部抽取项评估，结果写入 `extraction_result.hallucination_flag`。

**人工抽检校准**：每周从 LLM-as-judge 判定结果中随机抽 20 条人工复核，计算裁判准确率，若低于 0.85 则触发裁判Prompt优化。

#### 创新点2合规说明

幻觉防控方案与数据伦理/合规结合（详见 §15）：高风险抽取项不进入图谱、不用于匹配诊断，确保下游决策不受幻觉污染。

### 7.3 创新点3：岗位演进路径推荐

#### EVOLVES_TO关系发现

基于时间序列分析，发现岗位之间的演进关系：

- 对同一岗位在不同时间窗口的技能集进行差分分析
- 若岗位A在T2时期的技能集与岗位B在T1时期的技能集Jaccard相似度 > 0.6
- 且至少有3个独立数据源提供证据
- 则建立 EVOLVES_TO 关系: A -> B

#### 职业路径推荐算法

基于图谱EVOLVES_TO关系链，使用Cypher多跳查询推荐职业发展路径：
- 从当前岗位出发，遍历1-3跳的EVOLVES_TO关系链
- 按路径长度升序、证据强度降序排序
- 返回Top-5推荐路径
- 每条路径标注：演进时间窗口、需补充的关键技能

#### 前端展示

职业路径以"星座连线"方式在全景图谱中可视化：
- 演进时间窗口（平均X个月）
- 需要补充的关键技能
- 技能差距分析快速入口

### 7.4 创新点4：图谱质量仪表盘

#### 监控指标体系

| 指标类别 | 具体指标 | 计算方式 |
|---------|---------|---------|
| **规模指标** | 总节点数 | Cypher COUNT查询 |
| | 总关系数 | Cypher COUNT查询 |
| | 岗位覆盖率 | 已建图谱岗位数 / 目标岗位数 |

### 7.5 创新点5：能力通胀指数（可发布指标） ⭐ v2.0新增

赛题明确点名JD存在"通胀"现象（企业夸大技能要求）。现有方案仅在 §7.1 做了检测，但未将其升级为**可发布的、可比较的量化指标**。将通胀指数产品化，是技术创新突破性的体现。

#### 通胀指数定义

对每个岗位 P，定义岗位能力通胀指数（Capability Inflation Index, CII）：

```
CII(P) = median(skill_count of P's JDs in last 30 days)
       / baseline_skill_count(P)

其中 baseline_skill_count(P) 为该岗位的"基准能力要求"
基准 = 该岗位历史（>=6个月）JD 技能数的中位数
```

- CII = 1.0：当前要求与历史基准持平
- CII > 1.2：明显通胀（企业要求虚高）
- CII < 0.8：能力精简（可能反映工具成熟度提升）

#### 通胀校正在匹配中的应用

通胀 JD 会系统性压低求职者的匹配分数。匹配引擎（§6.3）对高 CII 岗位的技能要求做**通胀校正**：

```
对 CII > 1.2 的岗位：
  必备技能集合 = 按技能出现频率排序，取 Top-K（K = baseline_skill_count）
  频率低于 P30 的"边缘必备技能"降级为加分技能
```

这样求职者匹配分数更公平，更贴近真实岗位需求。

#### 可发布形态

- 全景图谱页展示每个岗位的 CII 徽章
- 演化看板展示 CII 时序曲线（岗位要求"膨胀/收缩"趋势）
- 输出"行业通胀报告"作为提交附件，体现洞察深度

### 7.6 创新点6：技能-岗位因果推理（突破共现局限） ⭐ v2.0新增

> ✅ **决策记录（D3）**：本创新点采用**轻量实现**——只实现核心的"条件独立性检验"简化版，作为"已探索"呈现，而非完整PC算法。通胀指数（§7.5）全做。理由：因果推理的完整PC算法工程量大，9人团队W12前完成风险高；轻量版既体现技术深度又控制风险。

传统图谱基于"共现"建立技能↔岗位关系（A和B经常一起出现→相关）。但**共现≠因果**：会Python和会做饭可能都常出现，但无职业关联。引入轻量因果推理，提升关系的可解释性，体现技术突破性。

#### 共现 vs 因果的区分

```
共现关系（correlation）：技能S与岗位P高频共现 → REQUIRES(P, S)
因果推理（causation）  ：技能S是岗位P"职责达成的必要手段" → REQUIRES(P, S) 标注为 causal
```

#### 因果推断方法（轻量版——条件独立性检验）

完整PC算法工程量过大，本项目实现其核心子集：

```
对候选关系 (P, S)，判断是否为因果而非纯共现：

1. 收集"控制变量集" Z = {P的其他高频技能, 行业, 级别}
2. 在控制 Z 后，测试 S 与 P 的条件独立性（用卡方检验或偏相关）：
   - 若条件独立 → 共现是被 Z 解释的虚假相关 → 降权 0.7×
   - 若仍条件依赖 → 可能存在直接因果 → 标注 causal=True，权重提升 1.2×
3. （轻量版不实现PC算法的后续方向确定步骤，仅做相关性vs因果性的区分）
```

#### 应用价值

- **匹配诊断可解释性**：告诉用户"这个技能是岗位核心因果要求，缺失影响大"，而非"恰好相关"
- **学习路径合理性**：因果链比共现链更适合指导学习顺序
- **新岗位定义质量**：新岗位的技能组合应具内在因果一致性，因果推理可过滤掉噪声共现

> 📌 **降级预案**：若 W12 前条件独立性检验仍未稳定，该创新点降级为"设计完成 + 实验性实现"，在文档中如实呈现探索过程与初步结果，不影响核心闭环与通胀指数（§7.5）。

## 十、测试方案

### 10.1 测试数据准备

| 数据类型 | 数量 | 来源 | 用途 |
|---------|------|------|------|
| JD原始数据 | >=500条 | BOSS直聘、拉勾、猎聘 | 全流程测试 |
| 标注JD数据（Golden Set） | 100条 | 人工双盲标注 | JD解析准确率测试 |

| 格式异常简历 | 10份 | 扫描件/加密/超长 | 简历健壮性测试 |

**Golden Set 质量门槛**：标注者间一致性 Cohen's Kappa >= 0.7，不达标的样本经仲裁人复核。

### 10.2 准确率测试方案（概要）

> 详细判定规则、边界case、评分脚本接口见 §14。

- **JD解析准确率**：100条标注JD，按字段分别计算 Precision/Recall/F1，加权汇总，目标 F1 >= 90%
- **简历提取准确率**：50份测试简历，技能归一化后逐字段比对，目标 F1 >= 90%（OCR降级样本单独报告）
- **人岗匹配准确率**：100个(JD,简历)配对，主协议为阈值二元判定准确率，辅报告 Top-K 与 Kendall Tau，目标 >= 90%

| 模块 | 覆盖率目标 | 重点测试项 |
|------|-----------|-----------|
| 信息抽取引擎 | >=70% | Prompt模板、JSON解析、后处理校验 |

## 十四、评估方法学与质量保障 ⭐ v2.0新增

> 本章是赛题"三项准确率≥90%"硬指标的可验证性核心。没有严谨的评估协议，准确率数字毫无说服力，评委也会质疑。

> 📌 **M2 评估里程碑（2026-06-19 达标）**：
> - 真实 MiMo API 端到端评估，50 条 Golden Set，F1 = **0.8767**（≥80% 达标）
> - Precision 0.898 / Recall 0.866
> - 评估脚本：`evaluation/run_real_eval.py`（增量保存+断点续跑+独立评分）
> - 双向归一化（抽取阶段 normalize.py 257别名 + 评分阶段 judge_eval.py 归一化）
> - 反作弊设计：Golden 真值不传抽取，仅在评分时对比（符合 D4 决策）
> - 三版本诚实对照：baseline 0.758（关键词匹配）/ 真实 0.8767（MiMo）/ LLM模拟 0.976（不可信）

### 14.1 评估方法学设计原则

| 原则 | 含义 | 反例（避免）|
|------|------|------------|
| **可复现** | 评委拿到脚本和Golden Set能跑出相同数字 | 依赖未公开的内部数据 |
| **口径明确** | "正确"的定义无歧义 | "差不多算对"的主观判定 |
| **防作弊** | 评估数据与训练/Prompt调优数据隔离 | 用Golden Set调Prompt再在Golden Set上报分 |
| **报告完整** | 不只报点估计，报置信区间与分项 | 只报一个笼统的"92%" |

### 14.2 Golden Set 设计规范

#### 14.2.1 数据集构成

| 数据集 | 数量 | 用途 | 标注Schema |
|--------|------|------|-----------|
| **JD-Golden** | 100条 | JD解析准确率 | job_title, core_responsibilities[], required_skills[], bonus_skills[], experience_years, education |
| **Resume-Golden** | 50份 | 简历提取准确率 | skills[]（含熟练度）, experience, education |

> ✅ **决策记录（D7）**：50份简历的组成 = **40份构造 + 10份脱敏真实 + 10份格式异常构造**（共60份，从中选50份评估用）。理由：构造简历可精准控制匹配度梯度（高/中/低）和格式多样性，是准确率测试需要的可控性；10份脱敏真实用于证明"系统在真实简历上也有效"，回应真实性质疑；脱敏真实部分严格走 §15 的PII脱敏流程。
| **Match-Golden** | 100对 | 匹配准确率 | (jd_id, resume_id, is_match: bool, match_strength: 0-1) |

#### 14.2.2 双盲标注与仲裁流程

```
每条样本 → 标注人A独立标注
        → 标注人B独立标注（双盲，互不可见）
              │
        A与B一致？ ─是─► 直接确认为真值
              │
             否
              │
        仲裁人C复核 → 裁定最终真值
```

**一致性门槛**：全数据集 Cohen's Kappa >= 0.7。低于此值说明标注标准不清，需重审标注规范而非强行通过。

**版本冻结**：Golden Set 确认后版本冻结（如 v1.0），存入 `evaluation_golden` 表，评估期间不得修改。任何Prompt/算法调优**不得**使用 Golden Set 数据，防止泄漏。

### 14.3 三项准确率的精确定义

#### 14.3.1 JD解析准确率

按字段分别评估，再加权汇总：

| 字段 | 判定方式 | 权重 |
|------|---------|------|
| job_title | 字符串归一化后精确匹配（去空格/大小写） | 0.15 |
| required_skills | 技能集 F1（归一化后比对，见下） | 0.30 |
| bonus_skills | 技能集 F1 | 0.20 |
| experience_years | 误差<=1年视为正确 | 0.15 |
| education | 精确匹配（含"不限"判定）| 0.20 |

**技能集 F1 计算**（核心）：
```
对抽取技能集 P 与真值技能集 G：
  1. 双方先做技能归一化（别名映射 + 向量相似度>=0.85 视为同一技能）
  2. TP = |P ∩ G|（归一化后交集）
  3. FP = |P - G|，FN = |G - P|
  4. 精确率 = TP/(TP+FP)，召回率 = TP/(TP+FN)
  5. F1 = 2*P*R/(P+R)
```

**边界case裁决**：
- LLM输出"机器学习"，真值为"深度学习" → 归一化后非同一技能 → 计 FP+FN（不正确）
- LLM输出"Python3"，真值为"Python" → 别名映射成功 → 计 TP（正确）
- 模糊匹配（相似度0.7-0.85）→ 标记 partial，**不计入正确也不计入错误**，单独报告 partial 比例

**加权汇总**：JD解析准确率 = Σ(各字段F1 × 权重)，目标 >= 90%。

#### 14.3.2 简历提取准确率

- 技能集 F1（同 §14.3.1 技能集计算）
- 分两层报告：**不含熟练度**（技能名匹配）与 **含熟练度**（技能名+熟练度均对才算TP）
- OCR降级样本（§5.4）单独报告，避免拉低或虚高主指标
- 目标：不含熟练度 F1 >= 90%（含熟练度为辅助参考，因主观性强）

#### 14.3.3 人岗匹配准确率

**主协议（阈值二元判定）**：
```
对每个 (jd, resume) 配对：
  系统输出 match_score ∈ [0,1]
  判定：match_score >= 0.6 → 预测"匹配"，否则"不匹配"
  与真值 is_match 比对 → 正确/错误
匹配准确率 = 正确判定数 / 总配对数（100对）
```

**辅报告（增强说服力）**：
- Top-K 命中率：对每个简历，目标岗位是否在系统推荐的 Top-3 内
- Kendall Tau：系统 match_score 排序与人工 match_strength 排序的相关性

**阈值论证**：0.6 阈值需用 Golden Set 校准——扫描 0.4~0.8，选准确率最高的阈值，并报告该选择的依据。

> ✅ **决策记录（D1）**：初始默认阈值定为 **0.6**。理由：综合匹配分（§6.3）已是 0.7×精确+0.3×语义 加权，0.6 能保证"明显匹配/不匹配"均正确分类，仅边界样本有歧义（而这些样本人工标注本身也模糊）。后续以 Golden Set 扫描校准结果为准，但 0.6 作为初始实现值。

### 14.4 评估脚本接口设计（可复现性）

```
输入：
  - golden_set.jsonl （冻结的标注数据）
  - system_output.jsonl （系统对相同样本的输出）
  - config.yaml （评估参数：阈值、归一化配置、权重）

输出：
  - report.json （各字段 P/R/F1、加权汇总、置信区间）
  - report.md （人类可读报告）

命令：
  python evaluate.py --golden golden_set.jsonl \
                     --output system_output.jsonl \
                     --config config.yaml
```

**评审复现保证**：Golden Set（脱敏后）、评估脚本、系统输出三者一并提交，评委可独立复现数字。

### 14.5 置信区间报告

点估计（如"F1=92%"）不够严谨，需报告 95% 置信区间：
- 采用 bootstrap 重采样（1000次）计算 F1 的 95% CI
- 报告格式：`JD解析 F1 = 92.3% [90.1%, 94.3%]`
- 若下限 < 90%，即使点估计达标也需说明风险

### 14.6 评估防作弊机制

> ✅ **决策记录（D4）**：Golden Set **100条全部作为 test set**，不切分 dev set。Prompt/算法调优时工程师**只看JD原文和抽取结果，不参考标注真值**。理由：① 满足赛题"≥100条JD及测试用例"的硬性要求，全做test统计显著性最强；② 不增加额外标注成本；③ 调优隔离真值既防泄漏又诚实。

| 作弊风险 | 防范措施 |
|---------|---------|
| 用 Golden Set 调 Prompt | **调优隔离真值**：Prompt工程师调优时只能看JD原文与系统输出，标注真值（ground_truth）由QA单独保管，仅在最终评估时解锁（D4决策）|
| 挑选 favorable 样本报分 | 必须对全量 Golden Set 报告，不得筛选 |
| 阈值过拟合 | 阈值校准基于"凭直觉调优"过程的经验，最终在 Golden Set（全test）上验证 |

**操作规范**：
- Golden Set 真值文件由 QA 专人保管，存放于独立目录，权限隔离
- 调优期间，算法组只能访问 JD 原文与系统输出 JSON
- 最终评估时，QA 运行评估脚本（§14.4），一次性输出报告，数字不可回溯性调参

| 数据类型 | 涉及法规 | 核心义务 |
|---------|---------|---------|
| 简历（含姓名、联系方式、经历）| 《个人信息保护法》(PIPL) | 知情同意、最小必要、可删除、脱敏处理 |
| 招聘JD（公开）| 《数据安全法》、反不正当竞争 | 仅采集公开数据、遵守robots.txt、不用于不正当竞争 |

### 15.2 简历数据合规处理流程

```
简历上传
   │
   ├─ [PII检测] 识别姓名、电话、身份证、邮箱等个人标识信息
   │
   ├─ [脱敏处理] PII替换为占位符（姓名→[NAME]，电话→[PHONE]）
   │
   ├─ [存储隔离] 脱敏后内容存业务库；原始文件加密存独立目录，访问审计
   │
   ├─ [LLM传输] 仅传输脱敏后内容至星火API（境内服务）
   │
   ├─ [保留期限] 默认保留30天，到期自动删除（compliance_log记录）
   │
   └─ [删除权] 用户可随时请求删除，7日内完成并记录
```

**关键技术实现**：
- PII检测：正则规则（电话/身份证/邮箱）+ NER模型（姓名/地址）

## 十七、协作工程规范（9人异步开发） ⭐ v2.0新增

> 9人异步开发是本项目最大的非技术风险——多数参赛团队并非倒在技术上，而是倒在协作混乱上。本章将 7 类高频协作问题转化为**强制纪律**，全员必须遵守。任何违反由技术负责人在迭代评审中通报并要求整改。

### 17.1 问题全景与规范总览

| 问题类别 | 典型灾难 | 强制规范 | 工具支撑 |
|---------|---------|---------|---------|
| ① 接口契约不同步 | 字段改名→前端白屏 | **契约优先**：先定契约再写代码 | OpenAPI + 自动生成双端类型 |
| ② 数据模型变更雪崩 | 加字段→图谱数据残缺 | **模型变更管制**：权限收归+迁移脚本 | Alembic + CHANGELOG |
| ③ 环境不一致 | 依赖版本差→结果不一致 | **Docker开发+依赖锁定** | docker-compose.dev + poetry.lock |
| ④ 任务依赖阻塞 | 流C等流B等流A→干等 | **Mock优先+接口冻结日** | MSW + 契约mock |
| ⑤ 代码合并冲突 | 2周不合→冲突几百处 | **Trunk-based+CI门禁** | GitHub branch protection |
| ⑥ 进度不可见 | 口头决定推翻文档决策 | **看板+决策日志+异步同步** | GitHub Projects + 附录D |
| ⑦ 集成灾难 | 各自跑通→合一起崩 | **每日集成+演示就绪** | CI全栈构建+冒烟测试 |

### 17.2 规范1：契约优先开发（接口契约不同步）🔴强制

**纪律红线**：
- ✅ **先定契约，再写代码**。W1-W2 必须产出接口契约 v1，经技术负责人签字后才允许动业务代码
- ✅ 接口契约是**单一事实源**（Single Source of Truth），存放于独立目录 `starmap-contracts/`
- ❌ 禁止在后端/前端代码里"随手改"接口字段，必须先改契约

**契约仓库结构**：
```
starmap-contracts/
├── openapi.yaml          # 所有REST API的权威定义（FastAPI自动生成+人工校验）
├── graph_cypher/         # 图谱查询接口（Cypher模板）

    ├── extraction.json   # JD/简历抽取结果Schema

```

### 17.3 规范2：数据模型变更管制（模型变更雪崩）🔴强制

**纪律红线**：
- ✅ 数据模型变更权限收归**技术负责人 + 算法组负责人**两人，其他人提需求不能直接改
- ✅ 所有变更必须走迁移脚本（Alembic），**禁止手动改库结构**
- ✅ 删除/改名字段必须保留旧字段**2周过渡期** + `@deprecated` 标记
- ❌ 禁止"我先改了再说，待会通知"

**变更通知机制**：
- 模型变更 PR 合并时，自动在协作群 @全体 发变更摘要（字段名、类型、影响范围）
- 维护 `CHANGELOG.md`，每次变更记录：日期、字段、增/删/改、影响模块、负责人

**本体版本管理**（Neo4j）：
- 本体定义带版本号（`ontology_v1.0`, `v1.1`...）
- 图谱节点存储其入图时的本体版本，便于追溯
- 本体不兼容变更需升版本号，并提供数据迁移脚本

### 17.4 规范3：Docker统一开发环境（环境不一致）🔴强制

**纪律红线**：
- ✅ **开发也用 Docker**（不只部署用），所有人通过 `docker-compose -f docker-compose.dev.yml up` 启动开发环境
- ✅ 依赖必须精确锁定：后端 `poetry.lock`，前端 `package-lock.json`
- ❌ 禁止 `pip install` / `npm install` 不加版本号
- ❌ 禁止"本地直装依赖"（除非Docker确实无法覆盖的特殊情况，需报备）

**dev 环境特性**：
```yaml
# docker-compose.dev.yml 关键特性
- 代码目录挂载（volume mount），支持热重载
- 后端：uvicorn --reload
- 前端：vite dev server HMR
- Neo4j/PostgreSQL/Redis/Chroma：与生产同版本
```

**环境漂移检测**：
- 远程服务器每日构建集成镜像，跑冒烟测试
- 若本地能跑、CI跑不通，**当天排查**，不积累"环境债"

### 17.5 规范4：Mock优先与接口冻结（任务依赖阻塞）🔴强制

**纪律红线**：
- ✅ W2 契约定稿后，**立刻**基于契约生成 mock 数据，消费方从 W3 起用 mock 开发，不依赖真实接口
- ✅ 流C（前端）用 MSW (Mock Service Worker) 拦截请求返回样例，完全独立于流B
- ✅ 每周一定为**接口冻结日**：本周不改已发布接口，改动下周二生效

**接口稳定性分级**：
| 级别 | 含义 | 消费方策略 |
|------|------|-----------|
| 🟢 稳定 | W2定死，原则上不变 | 可放心硬依赖 |
| 🟡 迭代 | 允许字段微调 | 预留容错（字段可选、默认值）|
| 🔴 实验 | 创新点接口，可能大改 | 用 feature flag 控制，不影响主流程 |

**关键解耦点**：流C从W3即可全速开发（用mock），流B的真实接口在W6联调时替换mock，两者并行不阻塞。

### 17.6 规范5：Trunk-based开发与CI门禁（合并冲突）🔴强制

**纪律红线**：
- ✅ **main 分支始终可运行**（最高纪律），禁止合入破坏运行的代码
- ✅ 分支生命周期 **<= 3天**，超过必须合并或向技术负责人说明
- ✅ 每天至少 `git pull origin main` 一次，把冲突解决在最小范围
- ✅ PR 必须 **CI全绿 + 至少1人code review** 才能合并
- ❌ 禁止直接 push main（分支保护强制）
- ❌ 禁止"攒两周再合并"（必然产生合并地狱）

**main 分支保护配置**：
- Require pull request before merging
- Require status checks to pass (CI: test/lint/typecheck)
- Require at least 1 approval
- 技术负责人持有 admin 权限兜底（紧急回滚）

**跨组 review 建议**：鼓励跨组 code review（如前端review后端接口实现），防止知识孤岛，但核心逻辑由本组专家review。

### 17.7 规范6：进度可视与决策持久化（进度不可见）🔴强制

**纪律红线**：
- ✅ 所有任务必须在 GitHub Projects 看板上流转，**禁止口头认领不落看板**
- ✅ In Progress 任务必须标"正常/阻塞"，阻塞项必须标"卡在哪、需要谁"
- ✅ 任何技术决策**不能只在群里说**，必须更新到文档附录D（决策记录）才算生效
- ❌ 禁止"群里决定了但文档没改"的决策漂移

**异步每日同步**（适合学生团队碎片时间）：
- 每人在看板任务卡更新：昨日完成 / 今日计划 / 阻塞项
- 替代部分站会，15分钟站会仅讨论阻塞项
- 技术负责人每天扫看板，阻塞项当天处理

**每周固定仪式**：
| 仪式 | 时间 | 内容 |
|------|------|------|

### 17.8 规范7：每日集成与演示就绪（集成灾难）🔴强制

**纪律红线**：
- ✅ **集成是每天的事**，不是W11才做的事
- ✅ 远程服务器每日拉 main，`docker-compose up` 全栈，跑端到端冒烟测试（E2E-1~4，§10.4）
- ✅ 任何环节挂了，**当天告警 + 当天修**，不留到第二天
- ✅ 从 W6 起，**main 分支必须随时能演示**（演示就绪纪律）

**契约测试（Consumer-Driven）**：
- 消费方写"我期望接口返回什么"的测试用例
- 提供方CI中跑这个测试，确保不破坏消费方
- 让"接口变了但没人知道"在CI阶段就暴露

**演示就绪纪律（Demo-Ready）**：
- 任何 PR 不能破坏演示种子数据加载（§16.5）
- CI 含"演示冒烟"任务：加载种子 → 首页能渲染 → 匹配页能跑 → 全绿才允许合并
- 这样到 W14 录视频时不会突然发现演示跑不起来

**集成联调周（W11-W12）特别安排**：
- 这两周**冻结新功能开发**（仅修bug + 集成）
- 全员专注联调，每天开"集成站会"专门报告"今天哪个接口对不上"
- 创新点（通胀/因果）若此时未完成，触发D3降级预案

### 17.9 协作规范执行保障

**规范的"牙齿"**（如何确保学生团队真执行）：
1. **CI 即执法者**：契约不一致、测试不过、覆盖率下降 → CI直接拒绝合并，无需人工监督
2. **看板即仪表盘**：技术负责人每日扫看板，阻塞/滞后项当天通报
3. **迭代评审即复盘**：周五走查规范执行情况，违规项记入会议纪要
4. **新人入职即培训**：任何成员加入第一天，必须读完本章并签字确认（协作群里回复"已读并遵守"）

**规范松绑机制**（避免教条）：
- 任何规范若经实践证明阻碍效率，可在迭代评审中提议修订
- 修订需技术负责人批准，更新本章并记录变更原因
- 但以下3条**永不可松绑**：① 契约优先 ② main可运行 ③ 每日集成
