# F1 优化方向（0.7900 → 0.80+）

> 基于 50 条 Golden Set 真实 MiMo 端到端评估的错误分析
> 日期：2026-06-17 | 作者：R7 李帅（流D QA）

## 当前状态

```
F1 = 0.7900  （M2 门禁 ≥0.80，差 0.01）
Precision = 0.8180
Recall = 0.7721

错误构成（214 个）：
  漏抽（false negative）: 129 (60%)  ← 主要瓶颈
  幻觉（false positive）:  82 (38%)
  误分类:                    3 (1%)
```

只需降低约 5 个错误即可达标。以下按投入产出比排序。

---

## P0：归一化增强（预期 +0.02~0.03，投入最小）

**问题**：抽取日志显示大量 `No normalization found for 'XXX'`，MiMo 抽到的技能与 golden 不匹配——不是抽错了，是别名表没覆盖。

**证据**（日志里的未命中技能）：
- `Celery` → golden 标的是 `Celery`，系统也抽到了，但 normalize.py 没归一化
- `Element Plus` / `SAP HANA` / `FFmpeg` / `GStreamer` / `MyBatis` / `Statistics`
- 这些技能在 golden 和系统输出中都存在，仅因拼写/别名差异被判为不匹配

**行动**：
1. 把 `skill_taxonomy.yaml`（198 技能）的 aliases 接入 normalize.py，替换硬编码 191 组
2. 重点补齐评估日志中出现的未命中技能
3. **预期效果**：归一化匹配率提升，直接消除约 15-20 个"假错误"，F1 → 0.81~0.82

**负责人**：杨博文（算法组，与本体 Issue #16 联动）

---

## P1：Prompt 漏抽优化（预期 +0.01~0.02）

**问题**：漏抽占 60%（129 个），是主要瓶颈。最差样本 jd-005（DevOps）漏抽 12 个技能，说明长 JD 的技能召回不充分。

**根因**：当前 Prompt 偏保守（反幻觉约束太强），MiMo 倾向少抽而非乱抽。

**行动**：
1. 调整 Prompt：增加"务必穷举 JD 中所有提及的技术名词"的引导
2. 对长 JD（>300字）做分段抽取，合并结果
3. 放宽反幻觉阈值（当前可能过于激进地删减技能）

**预期效果**：recall 从 0.77 → 0.82，F1 → 0.80~0.81

**负责人**：杨博文（算法组）

---

## P2：最差样本专项修复（预期 +0.005~0.01）

**问题**：Top 5 最差样本贡献了 54 个错误（占总错误 25%）。

| ID | 岗位 | 错误数 | 主要问题 |
|----|------|--------|---------|
| jd-019 | 安全工程师 | 17 | 安全领域技能表述模糊，MiMo 和 golden 标注口径不一致 |
| jd-005 | DevOps/SRE | 12 | 长 JD 技能多，召回不全 |
| jd-021 | 推荐算法 | 9 | 算法领域技能颗粒度不一致 |
| jd-012 | 大数据 | 8 | 同上 |
| jd-017 | 产品经理 | 8 | 软技能岗位，LLM 抽取困难 |

**行动**：
1. 检查这 5 条的 golden 标注是否合理（标注质量 vs 抽取质量）
2. 对安全/产品经理等"非标准技术岗"，Golden Set 的技能标注口径需要与 Prompt 对齐
3. 如果是 golden 标注问题 → 修正标注；如果是抽取问题 → Prompt 针对性优化

**负责人**：李帅（R7 QA，标注复核）+ 杨博文（抽取优化）

---

## P3：模型/Prompt 迭代（中长期）

**观察**：MiMo 是推理模型（每条 ~55 秒），推理开销大但抽取质量不错。

**可选方向**（如果 P0-P2 不够）：
1. 尝试 MiMo 不同温度参数（当前可能太保守）
2. 增加抽取 Prompt 中的 few-shot 示例
3. 对 SkillEntry 的 level/category 字段做后处理对齐
